# Алгоритм зовнішнього поширення

Коли ролі перетинають межі організацій через [обміни](/wiki/exchanges), обмеження поширення запобігає непередбаченому транзитивному витоку дозволів. Ця сторінка описує алгоритм та його обґрунтування.

## Проблема

Без обмежень поширення міжорганізаційний обмін ролями є транзитивним. Якщо організація A ділиться роллю з B, а B — з C, то дозволи A можуть витекти до C через B — хоча A і C не мають угоди.

Просте блокування всього зовнішнього успадкування занадто обмежує: це завадило б батьківській організації передати дозволи зовнішньої ролі дочірнім організаціям, що є природним адміністративним шаблоном.

## Вертикальна лінія

Ключове поняття — **вертикальна лінія** організації. Для організації X:

```
vertical_line(X) = ancestors(X) ∪ descendants(X)
```

Це включає саму X, усі організації вище (батьківська, прабатьківська, до кореня) та всі нижче (дочірні, онуки тощо). **Виключає** сусідів — інших дітей батька X.

```
        Root
       /    \
      A      B          vertical_line(B) = {Root, B, B1, B2}
     / \    / \          (виключає A, A1, D)
    A1  .  B1  B2

    D (окреме дерево)    vertical_line(D) = {D}
```

## Алгоритм

Обхід DAG відстежує один стан на шляху: `crossed_into` — організація, де вперше відбулося зовнішнє перетинання, або `null`, якщо перетинання ще не було.

### Правила

1. **Перетинання ще не було (`crossed_into = null`)**: перевіряємо, чи наступна роль належить до вертикальної лінії поточної організації.
   - **Так** (вертикальний рух): продовжуємо з `crossed_into = null`. Рух вгору до батька або вниз до дитини завжди вільний.
   - **Ні** (латеральний рух): це **зовнішнє перетинання**. Встановлюємо `crossed_into` на організацію наступної ролі.

2. **Вже перетнули (`crossed_into = X`)**: організація наступної ролі має бути на вертикальній лінії X. Якщо так — продовжуємо (зберігаючи `crossed_into = X`). Якщо ні — **блокуємо**: роль недосяжна цим шляхом.

### Псевдокод

```
queue = [(role, null) for role in user.direct_roles]
visited = {}

while queue is not empty:
    (role, crossed_into) = queue.pop()
    if (role, crossed_into) in visited: skip
    visited.add((role, crossed_into))
    collect(role)

    for child_role in role.inclusions:
        if crossed_into is not null:
            # Вже перетнули — залишаємось на вертикальній лінії crossed_into
            if child_role.org not in vertical_line(crossed_into):
                block
            else:
                enqueue (child_role, crossed_into)
        else:
            # Перетинання ще не було
            if child_role.org in vertical_line(role.org):
                enqueue (child_role, null)             # вертикальний рух
            else:
                enqueue (child_role, child_role.org)   # зовнішнє перетинання
```

## Приклад з розбором

### Налаштування

```
        Root Corp
       /         \
    Div A       Div B
   /              \
 Team A1        Team B1

 Ext D (окреме дерево, без батька)
```

**Обміни:**
- Div A ↔ Div B (двосторонній)
- Team B1 ↔ Ext D (двосторонній)

**Ролі та включення:**
- `Div B / Analyst` — має read на ресурсі Div B
- `Div A / Lead` включає `Div B / Analyst` (через обмін)
- `Div B / Analyst` включає `Team B1 / Intern` (вертикально — B1 є дитиною B)
- `Team B1 / Intern` включає `Ext D / Viewer` (через обмін)

**Користувач:** Аліса, в Div A, безпосередньо має роль `Div A / Lead`.

### Покрокова трасировка

| Крок | Роль | crossed_into | Дія |
|---|---|---|---|
| 1 | `Div A / Lead` | null | Збираємо. Починаємо з безпосередньої ролі. |
| 2 | → `Div B / Analyst` | ? | Div B не на vertical_line(Div A) = {Root, Div A, Team A1}. **Зовнішнє перетинання.** crossed_into = Div B. |
| 3 | `Div B / Analyst` | Div B | Збираємо. Перевіряємо включення. |
| 4 | → `Team B1 / Intern` | Div B | Team B1 на vertical_line(Div B) = {Root, Div B, Team B1}. **Дозволено.** crossed_into = Div B. |
| 5 | `Team B1 / Intern` | Div B | Збираємо. Перевіряємо включення. |
| 6 | → `Ext D / Viewer` | Div B | Ext D НЕ на vertical_line(Div B) = {Root, Div B, Team B1}. **Заблоковано.** |

**Результат:** Аліса отримує дозволи від `Div A / Lead`, `Div B / Analyst` та `Team B1 / Intern`. Вона **не** отримує дозволів `Ext D / Viewer`, хоча шлях через граф включень існує.

### Чому це важливо

Без обмеження поширення Аліса транзитивно отримала б дозволи Ext D через ланцюжок, з яким у неї немає прямої угоди. Правило вертикальної лінії гарантує: після перетинання на територію Div B, лише ієрархія Div B (вгору та вниз) доступна — без латеральних стрибків до неспоріднених організацій.

## Граничні випадки

### Обміни в одному дереві

Дві організації в одному дереві можуть мати обмін (наприклад, Div A ↔ Div B, обидві під Root Corp). Правило вертикальної лінії коректно це обробляє: рух від Div A до Div B — латеральний (Div B не є предком чи нащадком Div A), тому запускає зовнішнє перетинання.

### Декілька обмінів на одному шляху

Якщо шлях ролі перетинає декілька зовнішніх меж, лише **перше** перетинання встановлює `crossed_into`. Наступні рухи мають залишатися на вертикальній лінії першого перетинання. Це навмисно обмежувальне — запобігає ланцюгуванню через кілька зовнішніх організацій.

### Ланцюжок @members

Автоматичні ролі @members утворюють ланцюжок: дочірній @members включає батьківський @members. Це **вертикальний** рух (дитина → батько), тому ніколи не запускає зовнішнє перетинання. Дозволи @members завжди вільно поширюються вгору та вниз по дереву організацій.

## На масштабі

У продакшен-системі з ~10^3 організацій вертикальна лінія обчислюється з **матеріалізованого шляху**, що зберігається для кожної організації (див. [Архітектура](/wiki/architecture)):

```
vertical_line(X):
  target.path LIKE X.path || '%'     -- нащадки X
  OR X.path LIKE target.path || '%'  -- предки X
```

Це замінює рекурсивне обчислення предків/нащадків на індексоване порівняння рядків, роблячи перевірку поширення O(1) на ребро під час обходу DAG.
