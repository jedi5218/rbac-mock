# Архітектура для продакшену

Ця демонстрація працює як один процес із кешем у пам'яті. На продакшен-масштабі (~10^6 користувачів, ~10^5 ресурсів, ~10^4 ролей, ~10^3 організацій) архітектура розділяється на окремі шари з різними вимогами до продуктивності.

## Дві шляхи обробки

Всі операції належать до одного з двох шляхів з принципово різними вимогами:

### Шлях читання (перевірки доступу)

Гарячий шлях. Кожна дія користувача запускає перевірку: «чи може користувач X виконати дію Y над ресурсом Z?» Це має бути **O(1)** — один індексований запит, а не обхід графа.

Ефективні дозволи **попередньо обчислені** та зберігаються в матеріалізованій таблиці:

```
effective_permissions (user_id, resource_id, permission_bits)
```

Перевірка доступу зводиться до:

```sql
SELECT 1 FROM effective_permissions
WHERE user_id = :uid AND resource_id = :rid
  AND (permission_bits & :required_bits) = :required_bits
```

Кеш-шар Redis стоїть перед цією таблицею. При влучанні в кеш запит до бази не потрібен.

### Шлях запису (адміністративні операції)

Зміни графа ролей — додавання включення, зміна дозволів, призначення ролі — відбуваються рідко, але мають широку область впливу. Одна зміна включення ролі може торкнутися тисяч користувачів.

Операції запису:
1. Валідація (перевірка циклів, обмінів, правил поширення)
2. Запис до таблиць-джерел істини
3. Додавання завдання переобчислення в чергу
4. Фоновий воркер знаходить задіяних користувачів та оновлює матеріалізовану таблицю

Шлях запису має **кінцеву узгодженість**: існує короткий проміжок (секунди) між зміною ролі та оновленням ефективних дозволів. Це прийнятно, бо адміністративні операції рідкісні відносно перевірок доступу, а затримка обмежена.

## Ієрархія організацій на масштабі

При ~10^3 організацій у ~10-рівневій ієрархії рекурсивні запити (CTE) працюють, але є марнотратними для частих операцій: перевірок піддерев, пошуку предків та обчислення вертикальної лінії.

### Матеріалізований шлях

Кожна організація зберігає свій повний шлях від кореня:

| id | назва | шлях |
|---|---|---|
| 1 | Root Corp | `/1/` |
| 42 | Дивізіон A | `/1/42/` |
| 167 | Команда Alpha | `/1/42/167/` |

Типові запити стають рядковими операціями:

| Запит | SQL |
|---|---|
| X у піддереві Y? | `X.path LIKE Y.path \|\| '%'` |
| X — предок Y? | `Y.path LIKE X.path \|\| '%'` |
| Вертикальна лінія X | `X.path LIKE target.path \|\| '%' OR target.path LIKE X.path \|\| '%'` |
| Все піддерево | `WHERE path LIKE '/1/42/%'` |

Все індексовано. Все O(1) за пошук. Шлях оновлюється при переміщенні організації (рідко).

### Альтернатива: таблиця замикання

Зберігає всі пари (предок_id, нащадок_id, глибина) явно. ~10^3 організацій × 10 глибина = ~10^4 рядків. Підтримує ті ж запити через JOIN замість порівняння рядків.

## Переобчислення DAG ролей

Граф включення ролей має ~10^4 ролей з ~3 включеннями кожна в середньому (~30K ребер). BFS на користувача торкається ~50 ролей. Це прийнятно навіть при масовому переобчисленні.

### Інкрементальне переобчислення

Коли включення ролі змінюється (наприклад, роль R1 тепер включає R2):

1. **Знаходимо задіяних користувачів**: обходимо зворотний граф включень вгору від R1.
2. **Переобчислюємо для кожного**: запускаємо прямий обхід DAG (з правилами поширення) для кожного задіяного користувача.
3. **Оновлюємо матеріалізовану таблицю**: порівнюємо старі та нові ефективні дозволи, застосовуємо зміни.
4. **Інвалідуємо кеш**: видаляємо записи задіяних користувачів з Redis.

При ~1000 задіяних користувачів і ~50 ролей на обхід, пакет переобчислення — це ~50K відвідувань ролей — завершується за секунди у фоновому воркері.

## Стратегія інвалідації кешу

| Подія | Область інвалідації |
|---|---|
| Призначення/відкликання ролі користувача | Лише цей користувач |
| Зміна бітів дозволів ролі | Всі користувачі, що транзитивно мають цю роль |
| Додавання/видалення включення ролі | Всі користувачі, що транзитивно мають батьківську роль |
| Видалення ролі | Всі користувачі, що мали її |
| Створення/закриття обміну | Всі користувачі в піддеревах обох організацій |

**TTL як запобіжник**: всі записи кешу мають обмежений термін (наприклад, 1 година). Це обмежує вікно несвіжості навіть при пропущеній події інвалідації.

## Числа на масштабі

| Сутність | Кількість | Вплив на сховище |
|---|---|---|
| Організації | ~10^3 | Таблиця замикання: ~10^4 рядків |
| Користувачі | ~10^6 | user_roles: ~3M рядків (сер. 3 ролі/користувач) |
| Ролі | ~10^4 | Ребра включень: ~30K |
| Ресурси | ~10^5 | — |
| Ефективні дозволи | ~10^6 × ~20 ресурсів сер. | ~20M рядків |
| Переобчислення при зміні ролі | ~1K задіяних користувачів | Секунди у фоні |
| Латентність перевірки доступу | — | <1мс (влучання в кеш) / <5мс (запит до БД) |

## Що спрощує демо

| Аспект | Демо | Продакшен |
|---|---|---|
| Розв'язання дозволів | Обхід DAG на запит | Попередньо обчислена таблиця |
| Кеш | Словник Python у процесі | Кластер Redis |
| Запити ієрархії організацій | Рекурсивні CTE | Матеріалізований шлях або таблиця замикання |
| Переобчислення | Немає (обчислюється при читанні) | Фонові воркери з чергою завдань |
| Узгодженість | Миттєва (один процес) | Кінцева (обмежена затримка) |
| Горизонтальне масштабування | Один процес uvicorn | API без стану за балансувальником |
